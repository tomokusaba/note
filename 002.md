# エンジニアニメに寄稿する理由

結論から書くと、私がエンジニアニメに寄稿する理由は、**いつもの技術コミュニティとは違う文脈で技術の話を届けられ、違う前提を持つ人たちと会話できるから**です。

普段の私は、Microsoft 技術に関係するコミュニティで活動することが多いです。Azure、.NET、Microsoft 365、AI など、関心の近い人たちと深く話せる場はとても大事です。

一方で、技術コミュニティに長くいると、どうしても届く範囲が固定されやすくなります。同じ技術スタック、同じ課題感、同じイベントの空気感の中で話すことが増えていきます。

それは悪いことではありません。ただ、ときどき自分の発信を、もう少し違う場所に置いてみたくなります。

そのひとつが、エンジニアニメです。

もちろん、アニメが好きだから書いている、という面もあります。でも、それだけではありません。

[エンジニアニメ（技書博13サークルページ）](https://gishohaku.dev/gishohaku13/circles/1mjyQPU9mxTJvFFlncGr) は、アニメから得た学びをテーマに、技術者がそれぞれの記事を持ち寄るオムニバス本です。

「アニメと技術書」という組み合わせは、一見すると少し不思議に見えるかもしれません。

でも、実際に原稿を書いてみると、かなり自然でした。アニメには、労働、関係性、組織、アクセシビリティ、倫理、学び方など、技術者として考えたいテーマがたくさんあります。

## 今回寄稿した 3 つの記事

今回、私は次の 3 つの記事を寄稿しています。

- ギルドの受付嬢ですが、残業は嫌なのでボスをソロ討伐しようと思いますにみる労働構造
- 「クラスの大嫌いな女子と結婚することになった。」にみる関心と無関心
- 四月は君の嘘で読み解くWCAG──知覚・操作・理解・堅牢、4つの原則が届ける世界のかたち

この 3 本は、題材も技術テーマもかなり違います。

ただ、共通していることがあります。どの記事も、アニメの感想で終わらせるのではなく、そこから現実の仕事や IT の現場に戻ってくる記事にしています。

アニメを入口にして、抽象的なテーマを具体的な仕事の話へつなげる。

私は、そこにエンジニアニメで書く意味があると思っています。

## 労働構造を、ギルドという職場から考える

「ギルドの受付嬢ですが、残業は嫌なのでボスをソロ討伐しようと思いますにみる労働構造」では、ギルドをひとつの職場として読み替えています。

受付嬢の仕事は、依頼の受付、優先度付け、クレームの一次対応、日次レポートなどを担うフロントライン業務です。冒険者は実行部隊であり、ギルドは案件のマッチング、安全管理、事故対応を担う組織です。

そう整理すると、「残業が嫌だからボスをソロ討伐する」というタイトルは、単なる痛快な設定ではなくなります。

それは、窓口担当がいつの間にかインシデント対応の最前線になり、緊急案件が通常業務を侵食し、個人の能力と善意に組織が依存していく構造の話になります。

IT の現場でも、似たことは起こります。

サービスデスクが営業時間外の重大障害まで受け続ける。何でもできる人が深夜の緊急対応を一人で抱える。責任分界点が曖昧なまま、サイト信頼性エンジニアリング、プラットフォーム、アプリケーション、セキュリティの境界をまたいでしまう。

そういう状態を、原稿では「ソロ討伐」として捉えています。

大事なのは、ヒーローを称賛することではありません。

職務を言語化し、時間を可視化し、緊急時の例外運用を限定し、誰がどこまで責任を持つのかを明確にすることです。

「残業は嫌」はワガママではなく、健全な組織運営の第一歩です。

この話は、アニメだから軽く読めます。でも、扱っているテーマはかなり現実的です。だからこそ、エンジニアが読む意味があると思っています。

アニメの設定を通すことで、労働構造や責任分界点のような固い話題を、少し話し始めやすい言葉にできます。

## 「嫌い」を、対話と設計の材料にする

「『クラスの大嫌いな女子と結婚することになった。』にみる関心と無関心」では、「好き」の反対は「嫌い」ではなく「無関心」ではないか、というところから考えています。

北条才人と桜森朱音は、互いを苦手だと感じ、衝突を繰り返します。

でも、その衝突は、相手を見ているからこそ起こります。相手の言葉や行動に反応し、期待が裏切られたと感じ、苛立つ。そこには、歪んだかたちであっても関心があります。

本当に無関心であれば、衝突すら起きません。対話の入口も、修復のプロセスも立ち上がりません。

この構造は、IT の現場にもそのまま持ち込めます。

技術選定、設計方針、レビューコメント、特定の企業やプロダクトへの印象。そこでは「嫌い」という感情が出ることがあります。

たとえば「Microsoft だからだめ」「この言語は型がないから雑」「このフレームワークは信用できない」といった言い方です。

もちろん、そのままでは建設的ではありません。

ただ、その裏側には、過去の失敗経験、守りたい品質、運用負荷への不安、チームの学習コストへの懸念があるかもしれません。

だから原稿では、「嫌い」をただの拒絶ではなく、未調整のシグナルとして扱っています。

事実、解釈、感情、要望を分ける。何を優先するのかを言葉にする。小さく試し、数値で見直し、決めたことと見直し条件を記録する。

そうすれば、「嫌い」は対話と設計の材料になります。

これは、技術コミュニティにとっても大事な視点だと思っています。違う技術スタックの人と話すとき、最初に出てくるのは必ずしも好意だけではありません。違和感や警戒心から始まることもあります。

それでも、無関心で終わらせず、言葉にしていく。

エンジニアニメのような場では、アニメという共通の題材があることで、その対話を始めやすくなります。

## WCAG を、世界を届けるための設計として読む

「四月は君の嘘で読み解くWCAG──知覚・操作・理解・堅牢、4つの原則が届ける世界のかたち」では、WCAG の 4 原則を「四月は君の嘘」の物語と重ねています。

この原稿では、WCAG を単なるチェックリストとしては扱っていません。

「知覚可能」は、情報を受け取れるか。

「操作可能」は、行動できるか。

「理解可能」は、意味を正しく把握できるか。

「堅牢」は、それらが環境を問わず届くか。

この 4 つを、作品の中にある「届く」「届かない」という感覚と結びつけています。

有馬公生のモノトーンの世界は、知覚できないことの痛みとして読めます。音を失いながら鍵盤を叩き続ける指は、操作できることの尊さを示しています。かをりの手紙は、最後に意味が結びつくことで理解可能になる構造です。そして、楽譜という標準フォーマットは、どの環境でも音楽を届ける堅牢性の比喩になります。

Web アクセシビリティは、ときどき「対応しなければならないもの」として語られます。

もちろん、規格や達成基準は大事です。代替テキスト、キャプション、キーボード操作、言語指定、ARIA の名前と役割など、実装として押さえるべきことはたくさんあります。

でも、その根底にあるのは、誰かに情報を届けるための設計です。

情報は、存在するだけでは十分ではありません。知覚でき、操作でき、理解でき、環境を問わず届く形になって、初めて価値になります。

このように、アニメの物語を通すことで、技術仕様の奥にある考え方を説明できます。これも、エンジニアニメだからできる書き方だと思っています。

この 3 本を並べると、扱っているテーマは違っても、どれも「アニメを入口にして技術の話を開く」という共通点があります。だからこそ、普段の技術スタックの外側にも届きやすくなるのだと思います。

## 技術スタックの外側に届くこと

エンジニアニメに魅力を感じている大きな理由は、普段の技術イベントとは違う層に見てもらえることです。

IT 技術者といっても、技術スタックは本当にさまざまです。Microsoft 技術を中心に扱う人もいれば、AWS に詳しい人もいます。インフラ、セキュリティ、フロントエンド、バックエンド、データ、組織づくりなど、関心の置き方も違います。

普段のコミュニティでは出会いにくい人に、自分の記事が届く。

逆に、自分も普段あまり触れていない技術や考え方に出会える。

この相互作用が、とても面白いです。

Microsoft 技術のコミュニティにいると、どうしても Microsoft の文脈で話を組み立てがちです。もちろん、それは自分の軸ですし、大切にしているものです。

ただ、技術は Microsoft だけで閉じているわけではありません。

AWS に精通している人、別のクラウドを主戦場にしている人、まったく違う分野で設計や運用に向き合っている人。そういう人たちと話すと、自分の前提が少しずつ揺さぶられます。

その揺さぶりが、発信にも学びにも効いてきます。

## アニメは会話の入口になる

エンジニアニメの面白さは、共通の話題が「アニメ」であることです。

技術イベントでは、最初から技術スタックや専門領域が前面に出ることが多いです。

「Azure を使っています」

「AWS を使っています」

「セキュリティをやっています」

「アクセシビリティに関心があります」

そうした自己紹介は分かりやすい一方で、領域が違うと会話の距離が少し遠くなることもあります。

でも、アニメが間にあると少し変わります。

「その作品、見ました」

「その切り口で読むのは面白いですね」

「自分なら別の技術テーマで考えるかもしれません」

そうやって、技術の前に作品の話があります。だから、専門領域が違っても会話を始めやすいです。

共通のアニメという話題を通じて、技術について語れる。

私はそこに、エンジニアニメの大きな面白さを感じています。

## 技術の話は、技術だけから始めなくていい

技術記事を書くとき、つい「正確な仕様」「実装方法」「ベストプラクティス」から入らなければいけないと思ってしまうことがあります。

もちろん、それらは大事です。特に技術解説では、正確さは欠かせません。

ただ、技術について考える入口は、それだけではないと思っています。

ギルドの受付業務から、スコープクリープや責任分界点を考える。

嫌いから始まる関係から、技術選定やレビューにおける対話の設計を考える。

音が届かない物語から、WCAG の 4 原則とアクセシビリティの本質を考える。

そういう入口があってもいいと思います。

むしろ、普段は技術記事をあまり読まない人にも届く可能性があります。逆に、技術に詳しい人にとっても、いつもの概念を別の角度から見直すきっかけになります。

エンジニアニメは、そのためのよい場だと思っています。

## まとめ

私がエンジニアニメに寄稿するのは、アニメが好きだからだけではありません。

いつもの技術コミュニティとは違う層に届くこと。Microsoft 技術の文脈を越えて、別の前提を持つ人たちとも交流できること。共通のアニメという話題を通じて、技術について語れること。

そして何より、技術の話を技術だけから始めなくてもよい場であること。

そこに魅力を感じています。

アニメを入口にして、労働、関係性、アクセシビリティ、組織、設計、倫理について話してもいい。

そういう場があることで、技術コミュニティはもう少し広く、もう少し柔らかくなるのではないかと思います。

だから私は、エンジニアニメに寄稿しています。

そして今回のエンジニアニメは、技術書同人誌博覧会 13 に「い12」で出展します。

画像でも案内されているように、新刊を含めたエンジニアニメの各巻が並ぶ予定です。複数のエンジニアが「アニメから得た学び」をテーマに綴った本として、私の寄稿した 3 本もその中に入っています。

もしこの記事で少しでも気になったら、会場で実際に手に取ってもらえるとうれしいです。
