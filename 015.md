## Developer Blogs（.NET に興味がある人向け）

### Copilot は「どのモードがすごいか」より「どの仕事に使うか」

[Doing More with GitHub Copilot as a .NET Developer](https://devblogs.microsoft.com/dotnet/doing-more-with-github-copilot/?WT.mc_id=DT-MVP-5004827) は、.NET 開発者向けに GitHub Copilot の使い分けをかなり実務寄りに整理した記事でした。

この記事で一番大きいのは、**Copilot を機能一覧で理解するのではなく、仕事の種類で使い分ける**と整理している点です。  
たとえば、

1. サービスの責務を読む、リファクタ案を考える、失敗原因を言語化するなら chat
2. テスト追加、複数ファイルの更新、失敗の修正から再実行まで持っていくなら agentic workflows

という分け方です。

.NET 開発者目線では、Visual Studio、VS Code、Copilot CLI、cloud coding agent のどれを使うべきかが、かなり具体的に見えてきます。  
特に印象的だったのは、**「まず機能を覚える」のではなく、「今のタスクでどこを委譲するか」を考える**というメッセージでした。

これは実際の .NET 開発でもかなりしっくりきます。  
ASP.NET Core の既存サービスを読む、xUnit テストを増やす、OpenAPI と設定ファイルをまたぐ変更を入れる、`dotnet test` の失敗を CLI で詰める、といった日常の作業にそのまま落とし込めるからです。

「Copilot を使うかどうか」ではなく、**タスクに応じて chat と agent を切り替える**段階に入っている、と読むと分かりやすい記事でした。

### .NET MAUI 10 で Android を Material 3 に寄せやすくなった

[Give Your .NET MAUI Android Apps a Material 3 Makeover](https://devblogs.microsoft.com/dotnet/dotnet-maui-material-3/?WT.mc_id=DT-MVP-5004827) は、.NET MAUI 10 の Android アプリで Material 3 を有効にする話です。

ポイントはかなり明快で、**`net10.0-android` を含む .NET MAUI 10 プロジェクトなら、MSBuild プロパティを 1 つ足すだけで Material 3 を試せる**ようになっています。

```xml
<PropertyGroup>
  <UseMaterial3>true</UseMaterial3>
</PropertyGroup>
```

記事では、より広い範囲のコントロール対応を含む実用ラインとして、`Microsoft.Maui.Controls` **10.0.60 以降**が勧められていました。  
`Entry`、`SearchBar`、`DatePicker`、`Slider`、`ProgressBar`、`Button`、`Switch`、`Shell` などが対象で、Android 側の見た目を現行の Material Design にかなり寄せやすくなります。

大事なのは、これが**Android 専用の見た目改善**だという点です。  
iOS、Mac Catalyst、Windows はそれぞれのネイティブデザインのままで、既存の明示スタイルも優先されます。  
つまり、全面的な作り直しではなく、**既存アプリで安全に試しやすいアップデート**として見てよさそうです。

## GitHub Changelog（既存ユーザーへのインパクト重視）

### GitHub Advanced Security の予算を「超えないように止める」設定が入った

[Hard budget limits now available for GitHub Advanced Security](https://github.blog/changelog/2026-05-28-hard-budget-limits-now-available-for-github-advanced-security) は、既存の GHAS ユーザーほど効く更新です。

これまでは soft budget で通知はできても、実際には超過を止められませんでした。  
今回の更新で、**GHAS のライセンス予算に hard limit を設定し、上限到達後の新規ライセンス割り当てを実際に止められる**ようになっています。

影響が大きいのは、IdP グループ連携のようにライセンスが自動で増えやすい運用です。  
予算のつもりが実質アラートだけだった状態から、**本当に統制できる状態**に変わるので、Enterprise 管理者や billing manager にはかなり重要だと思います。

単なる課金まわりの話ではなく、**セキュリティ機能の展開速度とコスト統制をどう両立するか**に直結する更新でした。

### CodeQL 2.25.5 は GitHub Actions の検出精度改善が実務向き

[CodeQL 2.25.5 improves query accuracy for GitHub Actions](https://github.blog/changelog/2026-05-28-codeql-2-25-5-improves-query-accuracy-for-github-actions) は、一見地味ですが、既存の code scanning 利用者には効きます。

今回の更新では、GitHub Actions 向けに

1. `poisonable_steps` のモデリング拡張
2. `go run` や Python モジュール経由のスクリプト実行など、追加の sink 検出
3. `actions/unpinned-tag` が workflow だけでなく composite action の `action.yml` / `action.yaml` も解析

といった改善が入りました。

さらに、C/C++ や Java/Kotlin では誤検出を減らす修正も入っています。  
つまり今回は、**見つける範囲を広げつつ、ノイズも減らす**方向の更新です。

github.com の code scanning ユーザーには新しい CodeQL が自動展開されると書かれているので、既存ユーザーにとっては「いつか試す新機能」というより、**普段の検出結果の質がそのまま上がる更新**として見るのがよさそうです。

今回の更新は、派手な新製品発表の日ではありませんでした。  
ただ、.NET 開発者の視点で見ると、**Copilot の使い方がタスク起点で整理されてきたこと**と、**既存の GitHub 運用が少しずつ実務向けに強くなっていること**がよく分かる日だったと思います。

特に押さえておきたいのは次の4点です。

1. Copilot は「どの機能を触るか」ではなく「どの仕事を委譲するか」で使い分けると効果が出やすい

3. GHAS は予算通知だけでなく、超過を止める運用に進める
4. CodeQL は既存の code scanning 運用でも、検出精度の改善がそのまま効いてくる
