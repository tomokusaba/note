# note 記事執筆スキル

このスキルは、note に貼り付けやすい Markdown で記事を書くためのガイドラインです。既存の [`note-writing`](../../../.github/skills/note-writing/SKILL.md) スキルと、その Markdown 参照（[`markdown-elements.md`](../../../.github/skills/note-writing/references/markdown-elements.md)）を補完します。

Zenn 記事の作法を参考にしたオーケストレーションは使いますが、**note 固有の Markdown 形式は変更しません**。YAML フロントマター、Zenn CLI、`:::message` などの Zenn 固有記法は使いません。

---

## 1. 出力形式

### 1.1 基本

- 出力は note に貼り付けやすい Markdown にする。
- 記事タイトルは `#` またはプレーンテキスト見出しとして提示する。ユーザー指定がある場合はそれに従う。
- YAML フロントマターは追加しない。
- 公開状態、タグ、スラッグ、ファイル名などのプラットフォーム管理項目を勝手に本文へ追加しない。

### 1.2 ファイル作成

- 新規記事は、本文を書き始める前に連番の Markdown ファイルを新規作成する。
- 連番はリポジトリ内の既存 Markdown ファイル名から最大番号を確認し、次の番号を使う。例: `001.md`、`002.md`、`003.md`。
- 桁数は既存ファイルの規約に合わせる。規約がない場合は 3 桁ゼロ埋め（`001.md`）を使う。
- ユーザーが保存先ディレクトリを指定した場合はその配下で採番する。指定がない場合はリポジトリルート直下の Markdown ファイルを基準にする。
- 既存ファイルと衝突しないことを確認してから作成する。
- 作成した連番 Markdown ファイルに、note に貼り付けやすい完成稿を書き込む。

## 2. 記事の構成テンプレート

著者の記事は「導入 → 背景 / 問題 → 本論 → 具体例 → まとめ → 次の行動」の流れを基本にする。体験談、思考整理、How-to、技術解説で厚みを変える。

### 2.1 汎用テンプレート

```markdown
# タイトル

## はじめに
（何の記事か、誰向けか、読み終えると何が得られるかを 2〜4 段落で書く）

## 背景 / なぜこの話をするのか
（課題感、個人的な体験、読者が共感できる状況を置く）

## 本論
（主張・手順・考察を H2 / H3 で整理する）

### 具体例
（体験、比較、コード、引用、表などを必要に応じて置く）

## まとめ
（重要な持ち帰りを短く整理する）

## 次の行動
（必要な場合だけ、読者が試すこと・読むもの・考えることを書く）
```

### 2.2 How-to / 技術解説

- 「今回のゴール」「前提条件」「手順」「動作確認」「注意点」「まとめ」を軸にする。
- コードやコマンドはコピペしやすい粒度で提示する。
- 前提環境やバージョンを明記する。

### 2.3 体験談 / 思考整理

- 個人的な体験や違和感から入り、最後は読者にとっての示唆へ着地させる。
- 結論だけでなく、なぜそう考えたかの過程も残す。
- 重い話題では絵文字や軽い言い回しを控えめにする。

---

## 3. 文体・トーン

- 日本語で書く。
- 結論を先に出す。
- 段落は短めにする。
- 抽象論だけでなく、具体例・体験・比較を入れる。
- 過度に断定しない。未検証の内容は「執筆時点では」「公式ドキュメントでは」など出典に寄せた表現にする。
- 読者を急かさず、温かく、読みやすいトーンにする。
- 「いかがでしたでしょうか」のような定型句は避ける。

## 4. note Markdown 記法

既存の [`markdown-elements.md`](../../../.github/skills/note-writing/references/markdown-elements.md) を優先する。

### 4.1 強調

- 太字 `**...**` は本当に強調したい語句だけに使う。
- 取り消し線 `~~...~~` は、考えの変化や比較を示す場合に使う。

### 4.2 引用

- 引用 `>` は原典の一文、他者の意見、自分のメモを区別したい場合に使う。
- 外部情報を引用する場合は、引用直後に出典リンクを置く。

### 4.3 箇条書き・番号付きリスト

- 箇条書きは並列の要点に使う。
- 番号付きリストは手順・優先順位・時系列に使う。
- 1 項目だけ長すぎないように粒度を揃える。

### 4.4 リンク

- 外部リンクは `[テキスト](URL)` で書く。
- URL だけを雑に貼らず、なぜ参照するのかを前後の文で説明する。
- Microsoft Learn / Docs / DevBlogs / TechCommunity へのリンクには `WT.mc_id=DT-MVP-5004827` を付与する。

### 4.5 コードブロック

- コード・コマンド・設定例にはコードブロックを使う。
- 可能な限り言語名を付ける（例: `json`、`yaml`、`powershell`、`bash`、`csharp`、`typescript`）。
- 差分は `diff` を使う。

### 4.6 区切り線・配置・ファイル添付

- 区切り線 `---` は大きく話題が変わる箇所にだけ使う。
- 配置は読みやすさに明確な効果がある場合だけ使う。
- ワークシート、テンプレート、スライド、サンプルファイルが役立つ場合は、本文中に「ここにファイルを添付する」旨と説明を入れる。

---

## 5. 外部ドキュメント引用ルール

### 5.1 一次情報優先

- 技術仕様、API、バージョン、数値は一次情報を優先する。
- 個人ブログ、Qiita、Stack Overflow、note の他者記事は根拠にしない。参考にする場合も一次情報へ遡る。
- 確認できない内容は `<!-- TODO: 一次情報で要確認 -->` を残す。

### 5.2 Microsoft 系リンク

対象ドメイン:

- `learn.microsoft.com`
- `docs.microsoft.com`
- `devblogs.microsoft.com`
- `techcommunity.microsoft.com`

これらには `WT.mc_id=DT-MVP-5004827` を付与する。

```markdown
[Microsoft Learn の記事](https://learn.microsoft.com/dotnet/core/introduction?WT.mc_id=DT-MVP-5004827)
```

既にクエリ文字列がある場合は `&WT.mc_id=DT-MVP-5004827` を追加する。フラグメントがある場合はクエリをフラグメントより前に置く。

## 6. 執筆チェックリスト

- [ ] タイトルが本文の約束と一致している
- [ ] 導入で「誰向けか」「何が得られるか」が分かる
- [ ] 見出しが H2 / H3 中心で整理されている
- [ ] 各セクションが短い段落で読みやすい
- [ ] 具体例・体験・比較・手順のいずれかが入っている
- [ ] コードブロックに言語名が付いている
- [ ] 引用に出典が添えてある
- [ ] 画像やファイル添付が必要な箇所に説明がある
- [ ] Microsoft 系リンクに `WT.mc_id=DT-MVP-5004827` が付いている
- [ ] `<!-- TODO -->` が残っていない、または残す理由が明示されている
- [ ] note 固有の Markdown 形式を Zenn 形式へ置き換えていない

---

## 7. やってはいけないこと

- YAML フロントマターを勝手に追加する
- Zenn CLI や `articles/` 配置を前提にする
- `:::message` など Zenn 固有記法を使う
- note 固有の Markdown 形式を変更する
- 出典なしの数値・仕様を断定する
- 著者の文体を大幅に塗り替える
- 公開判断やタグ設定を勝手に決める
