### dotnet/runtime

#### WebAssembly と JIT の正しさ

1. **#130550 [browser] nodejs --experimental-wasm-exnref**

Node.js の実験的な WebAssembly exception references オプション `--experimental-wasm-exnref` を、WASM のドキュメント、サンプルのビルド設定、テスト起動用ターゲットへ広げる変更です。PR 本文の説明は「more places」にとどまるため詳細は限定的ですが、利用・検証経路で同じ実験的オプションを伝播させる意図が読み取れます。

[PR を見る](https://github.com/dotnet/runtime/pull/130550)

3. **#130543 [wasm] Fix CoreCLR P/Invoke EntryCount for libSystem.Globalization.Native**

ブラウザーと WASI 向けに生成される P/Invoke 呼び出しヘルパーのエントリ数が 33 のまま古くなり、34 件目の `GlobalizationNative_ToUnicode` に到達できず、`DllNotFoundException` が発生する問題を修正します。両プラットフォームの `callhelpers-pinvoke.cpp` を 34 件へそろえ、グローバリゼーションのネイティブ呼び出しテーブルを実装内容と一致させています。

[PR を見る](https://github.com/dotnet/runtime/pull/130543)

5. **#130446 JIT: always home register params on wasm**

WASM の JIT で生存中のレジスタ引数を常にホーム領域へ配置する前提に合わせ、`calleeRegArgMaskLiveIn` を初期化する変更です。呼び出し側で `this` を落とすケースの妨げにも対処しており、SuperPMI では 4 コンテキストで計 28 バイトの差分が報告されています。

[PR を見る](https://github.com/dotnet/runtime/pull/130446)

7. **#130426 Fix range-check monotonicity for multiplication**

JIT の範囲解析が、負のオペランドを含む乗算や左シフトまで単調増加とみなして比較を誤って畳み込む可能性があったため、拡張を加算に限定します。`rangecheck.cpp` の修正に加え、再発を検出する `Runtime_130425` 回帰テストを新設し、最適化の正しさを保ちます。

[PR を見る](https://github.com/dotnet/runtime/pull/130426)

#### WebAssembly の CI と実行環境

1. **#130511 [wasm] Bump helix V8 image and re-enable exnref CI legs (#129849)**

Linux の Helix WASM イメージを、exception references に対応する V8 15.2.47 へ更新し、停止していた Mono、CoreCLR、トリミング、パフォーマンスの CI レッグを再有効化します。複数のパイプライン YAML と `UpdateChromeVersions.cs` を変更し、Chrome/V8 更新処理には最低バージョン 15.1.103 のガードも加えています。

[PR を見る](https://github.com/dotnet/runtime/pull/130511)

#### Android の TLS 検証

1. **#124173 [Android] Respect platform trust manager in SslStream**

Android の `SslStream` がプラットフォームの `X509TrustManager` と `network_security_config.xml` を参照し、証明書ピンニングやカスタム信頼アンカーを検証へ反映するようにします。Java、ネイティブ、マネージドの SSL 実装とビルドタスクを更新し、実ハンドシェイクを使ったテストと機能テストを追加しています。

**破壊的変更**: Android のシステムストアにないマネージド専用のカスタム ルートは、従来 `None` だった場面でも `RemoteCertificateChainErrors` になることがあります。検証コールバックで受け入れることは可能であり、`ExtraStore` を使う `CustomRootTrust` はこの変更の影響を受けません。

[PR を見る](https://github.com/dotnet/runtime/pull/124173)

[PR を見る](https://github.com/dotnet/runtime/pull/130471)

[PR を見る](https://github.com/dotnet/runtime/pull/130473)

#### 実行時の安定性と性能

1. **#130526 Fix intermittent teardown crash in the cDAC GC stress framework**

cDAC GC ストレス基盤の終了時に、バックグラウンド検証とキャッシュ済み COM／リーダー状態の解放が競合し、ARM/ARM64 で SIGSEGV につながり得る問題を扱います。`cdacstress.cpp` と診断パイプラインを変更し、`s_cdacLock` を保持したまま初期化状態をクリア・再確認することで、解放済み状態へのアクセスを防ぎます。

ARM での CI 検証は引き続き必要ですが、x64 では 80 回の soak 実行を通過しています。

[PR を見る](https://github.com/dotnet/runtime/pull/130526)

3. **#130498 Make comhost cache COM activation delegate info**

`DllGetClassObject` のたびに hostfxr の読み込み、runtimeconfig の解析、ホストコンテキスト作成を繰り返していた comhost を見直します。`comhost.cpp` でアクティベーションデリゲートとロードコンテキストをプロセスの生存期間中に一度だけキャッシュし、繰り返す COM アクティベーションの準備コストを抑える変更です。

[PR を見る](https://github.com/dotnet/runtime/pull/130498)

5. **#130332 Arm64: use FEAT_CSSC cnt/ctz for PopCount and TrailingZeroCount**

Armv8.9／Armv9.4 で利用できる FEAT_CSSC の `cnt`／`ctz` 命令を、`PopCount` と `TrailingZeroCount` の JIT コード生成に使えるようにします。JIT、命令セット定義・設定・バージョン、CPU 機能検出を更新しており、ベンチマークは Apple M5 Pro で、とくに 64 ビット `PopCount` の改善を報告していますが、利用可否は対応ハードウェアに依存します。

[PR を見る](https://github.com/dotnet/runtime/pull/130332)
