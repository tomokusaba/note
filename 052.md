## Microsoft Developer Blogs（.NET に興味がありそうな新着）

### [Stop overloading your skills](https://developer.microsoft.com/blog/stop-overloading-your-skills)

*2026-06-27 (UTC)*

AI エージェント向けスキルの「肥大化」を戒める実践的なブログです。.NET SDK・ライブラリの開発者や Copilot 拡張機能を作っている方に直接刺さる内容です。

**主なポイント：**

- モデルはすでに膨大なドキュメントを学習済み。スキルが「すでに知っていること」を繰り返すと、有限なコンテキストウィンドウを無駄に圧迫する。
- まずスキルなしでシナリオを実行して「ベースライン」を測定する。モデルが正しく処理できる部分はスキルに含める必要はない。
- スキルのスコープは「モデルが失敗する部分だけ」に絞る。余分なトークンはドラッグ（抵抗）になり、他のスキルのパフォーマンスも落とす。
- **リーンスキルの作り方**：① シナリオ定義 → ② スキルなしで採点 → ③ 失敗箇所の特定 → ④ そのギャップだけカバー → ⑤ 再測定してトークンコストを確認。

## GitHub Changelog（既存ユーザーにインパクトが強い重要更新）

### [Track total merges by adoption phase in enterprise and organization reports](https://github.blog/changelog/2026-06-26-track-total-merges-by-adoption-phase-in-enterprise-and-organization-reports)

*2026-06-27 JST*

Copilot 使用状況メトリクス API の AI 採用フェーズ別コホートが強化され、Enterprise・Organization レポートに**マージ数の合計**が追加されました。

- `totals_by_ai_adoption_phase` の各エントリに `total_pull_requests_merged` が新たに追加。これまではユーザーあたりの平均値（`avg_pull_requests_merged`）のみでした。
- 採用フェーズごとの**絶対的なスループット**を比較・追跡できるようになり、AI 採用がデリバリー速度に与えるインパクトの測定が容易になります。
- 1日・28日のレポート両方で利用可能。Enterprise 管理者・Organization オーナーが Copilot 使用状況メトリクスにアクセスできる場合に有効。
