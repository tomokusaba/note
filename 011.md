# 2026年5月21日〜22日の .NET / GitHub 注目更新まとめ

## 先に結論

この 1 日分で先に読むなら、優先順は次の 3 つです。

1. [.NET 側] [Improving C# Memory Safety](https://devblogs.microsoft.com/dotnet/improving-csharp-memory-safety/?WT.mc_id=DT-MVP-5004827)
2. [GitHub 側] [Staged publishing and new install-time controls for npm](https://github.blog/changelog/2026-05-22-staged-publishing-and-new-install-time-controls-for-npm)
3. [GitHub 側] [Issue fields are now in public preview for all organizations](https://github.blog/changelog/2026-05-21-issue-fields-are-now-in-public-preview-for-all-organizations)

理由はシンプルで、今回は**将来の話に見えて、実は既存の開発や運用にそのまま効く更新**が多いからです。

.NET 側では、C# の `unsafe` が「危険な構文を許す印」から、**呼び出し側の安全責務を見える化する契約**へ寄っていきます。  
GitHub 側では、npm の公開フローと install 制御、Issue のメタデータ設計が、既存ユーザーの運用に直結します。

この記事は、.NET を触っている人、GitHub を日常的に使っている人向けに、2026年5月21日〜22日ごろの重要更新だけを短く整理したものです。

## .NET 側でまず押さえたい 2 本

### C# のメモリ安全性は、`unsafe` の意味そのものを見直しにきている

5月21日の [Improving C# Memory Safety](https://devblogs.microsoft.com/dotnet/improving-csharp-memory-safety/?WT.mc_id=DT-MVP-5004827) は、今回いちばん優先度が高い記事でした。

この投稿で面白いのは、`unsafe` を単なる構文上の切り替えではなく、**呼び出し側が何を守れば安全かを示す境界**として再設計しようとしている点です。  
unsafe な操作は inner `unsafe` ブロックの中に置き、必要ならその責務を呼び出し元へ伝播させ、さらに safety コメントで前提を明示していく構成が示されています。

実務目線で大きいのは、**どこが危険で、どこで責務を引き受けているのかをレビューしやすくする**方向だということです。  
AI 補助コード生成やレビュー支援が当たり前になるほど、「書ける」より「境界を読める」ことの価値が上がるので、この整理はかなり納得感があります。

時期感としては、.NET 11 で preview、.NET 12 で production を想定していて、最初は opt-in です。  
今すぐ全員のコードが変わる話ではありませんが、C# が安全性をどう扱おうとしているかを知るには先に読んでおきたい内容でした。

### MCP サーバーを .NET で作っているなら、ガバナンスを builder に載せやすくなる

同じく 5月21日の [Announcing Agent Governance Toolkit MCP Extensions for .NET](https://devblogs.microsoft.com/dotnet/announcing-agent-governance-toolkit-mcp-extensions-for-dotnet/?WT.mc_id=DT-MVP-5004827) は、対象読者は絞られますが、刺さる人にはかなり重要です。

これは Public Preview の `Microsoft.AgentGovernance.Extensions.ModelContextProtocol` を発表する記事で、対象は .NET 8+ と公式 MCP C# SDK です。

ポイントは、`IMcpServerBuilder` に **1 回の追加でガバナンスを組み込みやすい**ことです。  
記事では、startup scanning、policy enforcement、response sanitization、audit / metrics instrumentation がまとまって入る構成として説明されています。

特に実務的なのは、**fail closed が既定**だという点です。  
MCP サーバーを動かし始める段階で、「あとから安全策を足す」のではなく、最初から安全側の挙動を取りやすいのは大きいです。

すでに .NET で MCP サーバーを作っているチームなら、「どう作るか」ではなく「どう運用に耐える形にするか」の観点で読む価値があります。

## GitHub 側は既存ユーザーへの影響が大きい

### npm staged publishing は、公開フローを見直すきっかけになりそう

5月22日の [Staged publishing and new install-time controls for npm](https://github.blog/changelog/2026-05-22-staged-publishing-and-new-install-time-controls-for-npm) は、影響範囲の広さでかなり強い更新です。

npm CLI 11.15.0 以降で staged publishing が GA になり、CI から stage queue に publish したあと、**人間の maintainer が 2FA で承認して初めて公開される**流れが使えるようになりました。

この更新が効くのは、単に新機能だからではありません。  
trusted publishing を使っているチームでも、公開直前の proof of presence をより強く確保しやすくなるので、**既存の release workflow を安全側へ寄せる選択肢**になります。

もう 1 つ大きいのが install-time controls です。  
`--allow-file`、`--allow-remote`、`--allow-directory` が追加され、既存の `--allow-git` とあわせて、非レジストリ由来の依存取得を明示的に絞りやすくなりました。

しかも `--allow-git` の既定値は npm v12 で `none` に変わる予定です。  
つまりこれは「便利なオプションが増えた」で終わる話ではなく、**将来の既定値変更に備えて今の運用を確認しておくべき更新**だと思います。

### Issue fields は、成熟したチームほど効きそう

5月21日の [Issue fields are now in public preview for all organizations](https://github.blog/changelog/2026-05-21-issue-fields-are-now-in-public-preview-for-all-organizations) も、既存ユーザーにとってかなり実務的です。

org レベルの typed issue metadata が、github.com と data residency 付きの GitHub Enterprise Cloud で Public Preview になりました。  
型は single select、text、number、date をサポートし、検索やフィルタ、Project views、REST / GraphQL / webhooks でも使えます。

ここで大事なのは、「新しい項目を足せる」ことより、**Priority や Effort のような情報を label ではなく構造化データとして扱いやすくなる**ことです。

Issue 運用が成熟してくると、labels だけで優先度、種別、進行度、見積もりを表現するのが少し苦しくなります。  
Issue fields は、その詰まりを解消する方向の機能です。

3 月の preview 開始以降、すでに 1,000 を超える organizations が採用しているという点も印象的でした。  
既存の mature な運用に刺さっているからこそ広がっている、と読めます。

いま labels や Projects の設計に限界を感じているチームなら、かなり真面目に試す価値がありそうです。
