確認できた最近の .NET Blog 記事は、**2026年5月26日** のものまででした。

その一方で、周辺の Microsoft Developer Blogs には、.NET に関係が深い更新がありました。  
特に Azure Functions と Azure Cosmos DB は、**イベント駆動やサーバーレスを .NET で組む人ほど効く話**です。

GitHub 側では、Copilot の新規導入というより、**すでに使っている人の使い方が一気に広がった日**でした。  
アプリ、CLI、サンドボックスの 3 方向で、日常の使い方がかなり具体的になっています。

## Developer Blogs（.NET に興味がある人向け）

### .NET Blog 本体の新着はなし

執筆時点では、[.NET Blog](https://devblogs.microsoft.com/dotnet/?WT.mc_id=DT-MVP-5004827) 自体に、**過去 1 日の新しい記事は確認できませんでした**。

なので今日は、.NET 本体の直接の新着を追う日というより、**.NET 開発者に効く周辺の前進を拾う日**です。

### Azure Functions は「AI で楽になる」より「安全に今どきになる」が大きい

- [Introducing azure-functions-skills: An AI-Era Workspace for Azure Functions (Preview)](https://devblogs.microsoft.com/azure-sdk/introducing-azure-functions-skills-ai-era-workspace?WT.mc_id=DT-MVP-5004827)

今回の更新は、`azure-functions-skills` のパブリック プレビューです。  
npm パッケージとしては `@azure/functions-skills` が案内されています。

面白いのは、単に Azure Functions でも AI コーディング支援を使えるという話で終わっていないことです。  
記事では、**汎用のコーディング エージェントは Azure Functions に古い、あるいは安全でない実装パターンを出しがち**で、コード中にシークレットを書いてしまう例まで明記しています。

そのうえで、この仕組みはエージェントを次の方向へ誘導します。

- **Managed identity**
- **Key Vault references**
- **Flex Consumption**
- 新しいバインディングや同時実行パターン
- デプロイ前の **doctor** による検証

導入方法も実務寄りです。  
GitHub Copilot CLI、Claude Code、Codex CLI、VS Code 向けに、**1 コマンドで導入して使い始める流れ**が用意されています。

さらに初回セットアップ用のスキルで、前提条件のインストールやチェックを行い、チャットでは `functions-copilot` コンテキストを起動する構成になっています。  
つまり、個々の開発者が毎回ベストプラクティスを思い出すのではなく、**ツール側が足場を持つ**作りです。

.NET の読者目線で大事なのはここです。  
これは「AI コーディングがもっと楽になった」という話より、**Azure Functions 開発が、より安全で、より現行の推奨に寄った形で進めやすくなった**という変化として見るのがしっくりきます。

### Cosmos DB の change feed 強化は、Functions と組む .NET 設計に効く

- [Azure Cosmos DB All Versions and Deletes Change Feed Mode is Now Generally Available](https://devblogs.microsoft.com/cosmosdb/azure-cosmos-db-all-versions-and-deletes-change-feed-mode-is-now-generally-available?WT.mc_id=DT-MVP-5004827)

もうひとつ見逃せないのが、Azure Cosmos DB for NoSQL の **all versions and deletes change feed mode の GA** です。

このモードでは、作成・更新・削除・TTL 期限切れに加えて、**途中のバージョンも順序を保って取得**できます。

使いどころとして記事で挙げられているのは、次のような領域です。

- 監査やコンプライアンス
- soft delete に頼らないリアルタイム同期
- 削除を起点にしたワークフロー
- event sourcing

.NET 開発者の目線で特に大きいのは、**Azure Functions トリガーのサポートが .NET isolated worker と in-process の両方で利用可能**になったことです。

これは一見地味ですが、設計の自由度がかなり違います。  
Cosmos DB の変更を拾って Functions で処理する流れが、**削除や途中状態まで含めて組みやすくなる**ので、イベント駆動アプリの設計が一段やりやすくなります。

「更新が来た」だけではなく、**Cosmos DB と Azure Functions の組み合わせが、より素直に業務要件へ寄せられるようになった**と見ると、価値がわかりやすいです。

## GitHub Changelog（既存ユーザーへのインパクト重視）

### Copilot アプリは「使える人」が一気に増えた

- [Expanded technical preview availability for the GitHub Copilot app](https://github.blog/changelog/2026-06-02-expanded-technical-preview-availability-for-the-github-copilot-app)

GitHub Copilot アプリの technical preview は、**既存の Copilot Pro / Pro+ / Business / Enterprise 顧客すべて**へ広がりました。

今回の見出し役は **canvases** です。  
ただ、実務的にはそれだけではありません。Canvases は、作業の途中経過を見える形で置いて、編集や確認を同じ面でできる仕組みです。

あわせて案内されているのは、次の機能群です。

- voice conversations — 音声で Copilot に話しかけて操作や相談ができる機能です。
- cloud sessions — ローカルではなくクラウド上でエージェントセッションを回せる機能です。
- cloud automations — 定期実行や自動実行の作業をクラウドに任せる機能です。
- アプリ内の Copilot CLI sessions — CLI で始めたセッションをアプリ側でも追える機能です。
- agentic browsing — 統合ブラウザをエージェントが操作して、画面の確認まで進める機能です。
- rubber duck — いったん第三者目線で考えを点検するための、構造化された振り返り機能です。
- `/chronicle` — 過去の Copilot セッションから情報を引き直せる検索機能です。

ここでのポイントは、既存の有料ユーザーが **「そのうち来る未来の話」ではなく、今すぐ試せる機能や操作の幅がかなり広がった**ことです。  
Copilot の体験が、チャット 1 画面だけではなくなってきました。

### Copilot CLI は毎日触る道具としてかなり前進

- [Copilot CLI: Improved UI, rubber duck, prompt scheduling, and voice input](https://github.blog/changelog/2026-06-02-copilot-cli-improved-ui-rubber-duck-prompt-scheduling-and-voice-input)

CLI 側は、**既存ユーザーの体感がすぐ変わりそう**です。

まず、**rubber duck、prompt scheduling、voice input は今日から GA** です。

加えて、`/experimental on` で新しいターミナル UI が試せます。  
Issues、Pull requests、Gists のタブが入り、CLI のまま扱える情報の整理が進みました。

運用面で効きそうなのは、`/every` と `/after` です。  
セッション内でプロンプトやスキルをスケジュールできるので、**思いついたときに投げて、少し後に実行する**流れが作れます。

voice input が **ローカル環境で動く**と明記されているのも安心材料です。

派手さよりも、毎日の使いやすさです。  
Copilot CLI は「たまに触る実験ツール」から、**普段の作業導線に組み込みやすい道具**へ近づいています。

### サンドボックスの public preview は、導入の心理的ハードルを下げる

- [Cloud and local sandboxes for GitHub Copilot now in public preview](https://github.blog/changelog/2026-06-02-cloud-and-local-sandboxes-for-github-copilot-now-in-public-preview)

サンドボックスの public preview も大きい更新です。

ローカルでは `/sandbox enable` で有効化でき、**Microsoft MXC ベース**で、標準の Copilot seat に含まれるとされています。

クラウド側は `copilot --cloud` で、**隔離された一時的な Linux サンドボックス**を使えます。  
ポリシー継承、並列実行、オフロードされたタスク処理もポイントです。

これは、エージェント的な使い方を広げたい一方で、**隔離や統制は手放したくない**チームにかなり相性がよさそうです。

特に既存ユーザーにとっては、Copilot をより強く動かすための前提条件として、**「自分の作業環境をそのまま全面開放する」以外の選択肢も出てきた**と言えます。
