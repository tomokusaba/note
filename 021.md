# 2026年6月4日分：Foundry の agent 改良と Copilot SDK GA、既存ユーザー向け更新をまとめて確認

## 先に結論

執筆時点で、[.NET Blog](https://devblogs.microsoft.com/dotnet/?WT.mc_id=DT-MVP-5004827) の直近 1 日に新しい記事はありませんでした。  
その代わり、Microsoft Developer Blogs では **.NET で AI エージェントを作る人に効く更新**が 2 本並んでいました。

GitHub Changelog では、**既存ユーザーへの影響が大きい更新**が目立ちます。特に **Copilot SDK の GA** と **GPT-4.1 の廃止**は、使い方やモデル選定を見直すきっかけになりそうです。

## Developer Blogs（.NET に興味がある人向け）

### Foundry の agent optimizer は「作って終わり」から一歩進む

- [Introducing Agent Optimizer in Foundry Agent Service](https://devblogs.microsoft.com/foundry/agent-optimizer-build2026)

Foundry では、hosted agents を作って `azd deploy` で出すところまではかなり簡単になっていました。  
今回の agent optimizer は、その先にある **「本番でちゃんと振る舞うか」** を詰めるための機能です。

やっていることは、評価 → 改善案生成 → 再評価 → 比較 → 最適案のデプロイ、という閉ループです。  
改善対象は **instructions / skills / model / tool descriptions** で、プロンプト調整やツール定義の見直しを手作業で回す負担を減らします。

.NET でエージェントを組む人にとっては、コードを書く以前に **挙動の改善サイクルをどう回すか** が大事になります。  
この更新は、その足場を Foundry 側に持たせた、という見方がしっくりきます。

### Memory の強化は、エージェントの「覚える」を運用可能にする

- [Making agent memory more reliable, transparent, and production-ready](https://devblogs.microsoft.com/foundry/memory-build2026)

こちらは、memory を単なる会話の便利機能ではなく、**実行の再現性を上げる要素**として扱っている更新です。  
procedural memory によって、成功した手順や必要なチェックを再利用しやすくなり、同じ失敗を繰り返しにくくなります。

あわせて、Foundry portal には memory 管理 UI が入り、TTL、multimodal support、direct memory command も追加されました。  
さらに Microsoft Agent Framework では、**markdown ファイルベースの memory** も案内されています。

記事では、procedural memory を有効にすると **STATE-Bench と Tau-Bench で約 5% の改善**が見えているとも述べられています。  
AI エージェントは「賢さ」よりも、まず **同じ手順を安定して再現できるか** が本番では効くので、かなり実務寄りの更新です。

## GitHub Changelog（既存ユーザーへのインパクト重視）

### Copilot SDK が GA になった

- [Copilot SDK is now generally available](https://github.blog/changelog/2026-06-02-copilot-sdk-is-now-generally-available)

これは .NET 読者にもかなり刺さる更新です。  
Copilot の agentic engine を、自分のアプリやツールに **安定 API で組み込める** ようになりました。

対応言語には **.NET** も含まれていて、`dotnet add package GitHub.Copilot.SDK` で使い始められます。  
MCP、カスタムツール、hooks、OpenTelemetry tracing、cloud sessions まで含まれているので、単なるチャット API ではありません。

既存ユーザー目線では、Copilot を「使う」だけでなく、**Copilot を部品として埋め込む** 選択肢が現実的になったのが大きいです。

### GPT-4.1 の廃止は、モデル運用を見直すサイン

- [GPT-4.1 deprecated](https://github.blog/changelog/2026-06-02-gpt-4-1-deprecated)

Copilot の各体験で **GPT-4.1 が廃止**され、代替として **GPT-5.5** が案内されています。  
個人利用でも組織利用でも、いつものモデルがそのまま使える前提ではなくなりました。

特に管理者は、model policy 側で代替モデルを有効にしておく必要があります。  
日常利用では気づきにくくても、**自動で選ばれていたモデルが変わる** ことは体感差につながるので、早めの確認が無難です。

### Copilot app と VS Code 側も、既存ユーザーの作業面が広がった

- [Expanded technical preview availability for the GitHub Copilot app](https://github.blog/changelog/2026-06-02-expanded-technical-preview-availability-for-the-github-copilot-app)
- [GitHub Copilot in Visual Studio Code, May releases](https://github.blog/changelog/2026-06-03-github-copilot-in-visual-studio-code-may-releases)

Copilot app の technical preview は、既存の Pro / Pro+ / Business / Enterprise に広がりました。  
canvases、voice conversations、cloud sessions、agentic browsing など、**エージェント前提の作業面**がかなり増えています。

VS Code 側でも、Agents window の preview、remote agents、session sync、BYOK の air-gapped 対応など、既存ユーザーの使い方を押し広げる更新がまとまっていました。  
普段から Copilot を使っている人ほど、**「どの画面で何をやるか」** を見直す価値があります。

## まとめ

- **Developer Blogs では、Foundry の agent optimizer と memory 強化が .NET/AI エージェント開発に効く**
- **GitHub 側は、Copilot SDK GA と GPT-4.1 廃止が特に重要**
- **Copilot app と VS Code の更新で、既存ユーザーの作業導線も広がった**

今日は新機能の話というより、**本番運用と既存ユーザーの前提が少しずつ変わる日** でした。  
AI エージェントを触っているなら、Foundry と Copilot の両方を一度見直しておくのがよさそうです。
