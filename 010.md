# 今週の .NET / GitHub 重要更新まとめ

## 先に結論

今週は、派手な新機能というより、**普段の開発や運用の前提が少し変わる更新**が多めでした。

優先度だけ先に挙げると、まず見るなら次の 3 本です。

1. [.NET 側] [Improving C# Memory Safety](https://devblogs.microsoft.com/dotnet/improving-csharp-memory-safety/?WT.mc_id=DT-MVP-5004827)
2. [.NET 側] [NuGet Package Pruning: Cleaner Dependencies and Actionable Vulnerability Reports](https://devblogs.microsoft.com/dotnet/nuget-package-pruning-in-dotnet-10/?WT.mc_id=DT-MVP-5004827)
3. [GitHub 側] [Auto model selection now routes based on your task in VS Code](https://github.blog/changelog/2026-05-20-auto-model-selection-now-routes-based-on-your-task-in-vs-code)

理由はシンプルで、どれも**一部の先行ユーザーだけの話ではなく、既存の開発フローにそのまま効いてくる**からです。

この記事は、.NET 開発者、GitHub / Copilot を日常的に使っている人、技術動向を短時間で追いたい人向けに、今週の重要更新を短く整理したものです。  
全部を追う時間がなくても、「何を先に読むべきか」が分かるように整理します。

## .NET 側は「これからの前提」を押さえる週

### まずは C# の安全性の考え方を見ておきたい

[Improving C# Memory Safety](https://devblogs.microsoft.com/dotnet/improving-csharp-memory-safety/?WT.mc_id=DT-MVP-5004827) は、今週の .NET 側でいちばん方向性が大きい話だと思いました。

この投稿では、`unsafe` を単なるポインター構文の入口ではなく、**呼び出し側にどんな安全性の責務があるかを示す契約**として見直そうとしています。  
あわせて安全性コメントの書式を導入し、どこに注意が必要かをレビューしやすくしていく流れも示されています。

執筆時点の説明では、.NET 11 でプレビュー、.NET 12 で本番利用を想定し、初期はオプトインの位置づけです。  
特に印象的だったのは、AI 補助コード生成が増えるほど「書けること」よりも**レビューしやすいこと**が重要になる、という文脈です。

普段 `unsafe` を使わない人でも、C# が安全性をどう言語化しようとしているかを知る意味で、先に読んでおく価値がありそうです。

### NuGet の pruning は、警告の見え方を変える更新

[NuGet Package Pruning: Cleaner Dependencies and Actionable Vulnerability Reports](https://devblogs.microsoft.com/dotnet/nuget-package-pruning-in-dotnet-10/?WT.mc_id=DT-MVP-5004827) は、実務でじわっと効くタイプの更新です。

.NET 10 では `NuGetAuditMode=all` が既定となり、transitive auditing が有効になります。  
そのうえで platform-provided package を restore graph から pruning することで、**依存関係のノイズや脆弱性レポートの偽陽性を減らす**方向に進んでいます。

公式ブログでは、transitive vulnerability report が 70% 減少したというテレメトリにも触れられていました。  
この数字そのもの以上に大事なのは、警告が増える話ではなく、**本当に見るべき警告を見やすくする話**だという点です。

CI で監査を回しているチームや、依存関係の警告が多くて判断しづらくなっていた人ほど、優先して読んでよさそうです。

### Process API の改善は、CLI 連携を書く人にかなり効きそう

[Process API Improvements in .NET 11](https://devblogs.microsoft.com/dotnet/process-api-improvements-in-dotnet-11/?WT.mc_id=DT-MVP-5004827) も見逃しにくい更新でした。

`Process.RunAndCaptureText[Async]`、`Run[Async]`、`StartAndForget`、`ReadAllText[Async]` などが入り、これまで少し面倒だったプロセス実行周りがかなり整理されています。  
stdout / stderr を同時に読み出しやすくし、デッドロックを避けやすくしている点も実務的です。

さらに `KillOnParentExit`、`StartDetached`、`SafeProcessHandle` なども追加され、性能面でも Apple Silicon や Unix 系での改善が説明されています。  
外部コマンド実行、ツール連携、ビルド補助を書くことが多い人には、早めに見ておきたい内容です。

### 全体像を追うなら Preview 4 が入口になる

個別記事に入る前に見取り図がほしいなら、[.NET 11 Preview 4 is now available!](https://devblogs.microsoft.com/dotnet/dotnet-11-preview-4/?WT.mc_id=DT-MVP-5004827) が便利です。

Process API の話だけでなく、MCP Server template SDK の同梱、`dotnet watch` の改善、EF Core の vector search など、今週の広い流れを一度につかめます。  
「まず全体、その後に深掘り」という読み方をしたい人向けです。

### MAUI と AI / MCP は対象者ならチェック

[.NET MAUI Moves to CoreCLR in .NET 11](https://devblogs.microsoft.com/dotnet/dotnet-maui-moves-to-coreclr-in-dotnet-11/?WT.mc_id=DT-MVP-5004827) は、MAUI を使っている人にとって優先度が高い話です。

Android / iOS / Mac Catalyst / tvOS で CoreCLR が既定になり、ランタイム統一が進みます。  
一方で、公式記事でも大規模アプリでは回帰報告に触れているので、期待だけでなく**実測前提で判断する更新**として読むのがよさそうです。

AI / MCP 側では、[Announcing Agent Governance Toolkit MCP Extensions for .NET](https://devblogs.microsoft.com/dotnet/announcing-agent-governance-toolkit-mcp-extensions-for-dotnet/?WT.mc_id=DT-MVP-5004827) も興味深いです。  
ポリシー適用やレスポンスのサニタイズなどを既存の builder パイプラインに組み込みやすくする、という整理なので、MCP サーバーを作るだけでなく運用ガバナンスまで考えたい人には刺さりそうです。

## GitHub 側は「既存ユーザーへのインパクト」で見る

### いちばん広く効きそうなのは Auto model selection

[Auto model selection now routes based on your task in VS Code](https://github.blog/changelog/2026-05-20-auto-model-selection-now-routes-based-on-your-task-in-vs-code) は、Copilot を普段使っている人ほど影響が大きい更新です。

Auto が、タスク内容、利用率、モデルの健全性などを見て自動でルーティングし、管理者ポリシーも尊重するようになります。  
しかも、使い方を変えなくても体験が変わるタイプの更新です。

これは「新しい機能を覚える」話ではなく、**いつもの Copilot の中身が変わる**話です。  
日常利用している人ほど、回答の傾向や速度感の違いを感じる場面が出てきそうです。

### Issue fields は、ラベル運用を見直すきっかけになりそう

[Issue fields are now in public preview for all organizations](https://github.blog/changelog/2026-05-21-issue-fields-are-now-in-public-preview-for-all-organizations) も、既存ユーザーへの影響が大きい更新です。

single select、text、number、date の型付きメタデータを組織レベルで issue に付けられるようになり、search / filter、project column、timeline、REST / GraphQL / webhook にも対応しています。

つまり、Priority や Effort のような情報を、従来の label だけで持つのではなく、**structured metadata として扱いやすくなる**ということです。  
Issue 運用が育っているチームほど、「今のラベル設計をそのまま続けるか」を考える材料になりそうです。

### npm staged publishing は、公開とインストールの前提に触る

[Staged publishing and new install-time controls for npm](https://github.blog/changelog/2026-05-22-staged-publishing-and-new-install-time-controls-for-npm) は、npm を配布・運用しているチームにはかなり重要です。

npm CLI 11.15.0 以降で staged publishing が GA となり、`npm stage publish` と 2FA 承認で proof of presence を強化する流れが出ています。  
さらに `--allow-file`、`--allow-remote`、`--allow-directory` が追加され、`--allow-git` の既定も将来的に v12 で `none` へ変わる予定とされています。

ここは新機能というより、**公開フローと依存取得の安全側への調整**として見るのが分かりやすいです。  
既存のスクリプトや運用手順に影響しそうなら、早めに確認しておいた方が安心です。

### Dependabot の Python 3.9 非対応予告は、運用上かなり実務的

[Upcoming deprecation of Python 3.9 for Dependabot](https://github.blog/changelog/2026-05-19-upcoming-deprecation-of-python-3-9-for-dependabot) は、地味ですが見落としにくい更新です。

2026-06-23 以降、Python 3.9 は Dependabot の対象外になり、継続利用すると依存更新 PR が作成されなくなる可能性があります。  
これは「いつか対応する」より、**止まる前に気づくべき運用情報**に近いです。

Dependabot を任せている前提があるなら、対象リポジトリのバージョン確認は早めにやっておくのがよさそうです。

### Copilot 周辺は「探す」「直す」が少しずつ前進

[Semantic issue search in Copilot Chat](https://github.blog/changelog/2026-05-20-semantic-issue-search-in-copilot-chat) では、Copilot Chat on web から自然言語で issue を意味検索できるようになりました。  
計画立案やトリアージの初動で、「似た issue があったはず」を探しやすくなる更新です。

[Easily apply Copilot code review feedback with Copilot cloud agent](https://github.blog/changelog/2026-05-19-easily-apply-copilot-code-review-feedback-with-copilot-cloud-agent) では、`Fix with Copilot` としてレビュー指摘の反映フローが強化され、複数コメントの一括修正にも対応しています。

どちらも即座に全員の働き方を変えるほどではないかもしれません。  
ただ、Copilot が補完だけでなく、**探す・整理する・修正する**ところまで少しずつ広がっている流れは押さえておきたいです。

## まとめ

今週の更新を短くまとめると、こうです。

.NET 側は、**C# の安全性、NuGet の監査ノイズ削減、Process API の実用改善**が中心でした。  
GitHub 側は、**Copilot の既定体験、Issue 管理、npm / Dependabot の運用前提**に効く更新が目立ちました。

全部を追う時間がないなら、優先順はこの並びがおすすめです。

まずは [Improving C# Memory Safety](https://devblogs.microsoft.com/dotnet/improving-csharp-memory-safety/?WT.mc_id=DT-MVP-5004827)、  
次に [NuGet Package Pruning: Cleaner Dependencies and Actionable Vulnerability Reports](https://devblogs.microsoft.com/dotnet/nuget-package-pruning-in-dotnet-10/?WT.mc_id=DT-MVP-5004827)、  
GitHub 側は [Auto model selection now routes based on your task in VS Code](https://github.blog/changelog/2026-05-20-auto-model-selection-now-routes-based-on-your-task-in-vs-code) を先に読むと、今週の変化の輪郭がつかみやすいと思います。

派手な発表より、普段の前提を少し変える更新ほど後から効いてきます。  
今週はまさに、そのタイプの週でした。
