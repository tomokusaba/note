# 2026年6月25日分：Developer Blogs（.NET 関連）と GitHub Changelog の重要更新

## Microsoft Developer Blogs（.NET に興味がありそうな新着）

### [When the model has never seen your code](https://devblogs.microsoft.com/blog/when-the-model-has-never-seen-your-code)

*Waldek Mastykarz 著 / 2026-06-24 (UTC)*

Agent Experience (AX) シリーズの第5回。モデルがあなたの技術を全く知らない（学習データにない）場合の AI コーディングエージェントとの付き合い方を解説しています。

**ポイント**:

- モデルは「知らない」と言わず、最も近いパブリック技術にマッピングして**それらしいが誤ったコードを生成する**（Closest-match trap）
- 社内SDKや独自フレームワークでは、エージェント拡張の役割が「誤りの訂正」から「ゼロからの教育」に変わる
- 効果的な教育の順序は **① 技術の正体と目的 → ② コアコンセプト → ③ API形状と慣習 → ④ 典型パターン → ⑤ エッジケース** の階層を守ること。多くのチームは逆から始めてしまい失敗する
- `AGENTS.md` への「これは○○ではなく××だ」という明示的な否定文が closest-match を上書きするうえで効果的
- MCP サーバーでオンデマンドに API リファレンスを提供することで、全文をプロンプトに埋め込むトークンコストを避けられる
- ベースライン評価（拡張なしで動かす）でモデルが何の技術だと誤認しているかを把握してから拡張を設計する

.NET で社内ライブラリや独自フレームワークを持っている組織が GitHub Copilot を活用する際に、そのまま実践できる設計指針です。

## GitHub Changelog（既存ユーザーにインパクトが強い重要更新）

### [Changes to model selection for Free and Student plans](https://github.blog/changelog/2026-06-24-changes-to-model-selection-for-free-and-student-plans)

*2026-06-24 (UTC)*

**Copilot Free・Student プランのモデル選択が「自動」一択に変更**されました。

- 今後はタスクに応じて最適なモデルを自動選択する **Copilot auto** がデフォルトかつ唯一の選択肢になります
- 自動選択はプランの制限内で複数のモデルファミリーにアクセスできる設計です
- あわせて、Microsoft リリースモデルについて付いていた `(Preview)` ラベルが廃止されました。auto が裏でルーティングを管理するため、ユーザーがラベルを判断材料にする必要がなくなったためです

Free・Student プランで特定モデルを指定して使っていたユーザーは、今後その選択肢がなくなる点に注意が必要です。
