# 今週気になった .NET / GitHub の更新まとめ
## 先に読むべきものから、短時間で追えるように整理しました

## はじめに

先に結論を書くと、今週の更新は**「新機能が増えた」よりも、「これからの開発や運用の前提が少し変わる」話が多かった**です。

.NET 側では、C# の安全性の考え方、NuGet の依存関係整理、Process API の使い勝手改善、MAUI のランタイム統一など、今後の開発体験にじわっと効いてくる更新が目立ちました。  
GitHub 側では、npm の公開フロー、Issue 管理、Copilot の使い方が、既存ユーザーにも影響しそうです。

この記事は、.NET 開発者、GitHub/Copilot を使っている人、技術動向を短時間で追いたい人向けです。  
「全部は読めないけれど、何を先に読むべきかは知りたい」という前提で、優先度が分かる形でまとめます。

## .NET 側の注目点

### まず見るなら、この 3 本

今週の .NET 側で、まず読んでおく価値が高いと感じたのは次の 3 つです。

1. [Improving C# Memory Safety](https://devblogs.microsoft.com/dotnet/improving-csharp-memory-safety/?WT.mc_id=DT-MVP-5004827)
2. [NuGet Package Pruning: Cleaner Dependencies and Actionable Vulnerability Reports](https://devblogs.microsoft.com/dotnet/nuget-package-pruning-in-dotnet-10/?WT.mc_id=DT-MVP-5004827)
3. [Process API Improvements in .NET 11](https://devblogs.microsoft.com/dotnet/process-api-improvements-in-dotnet-11/?WT.mc_id=DT-MVP-5004827)

理由はシンプルで、どれも**特定の一部ユーザー向けの話ではなく、普段の C# / .NET 開発の前提に関わるから**です。

### C# の `unsafe` を「責務の契約」として見直す話

[Improving C# Memory Safety](https://devblogs.microsoft.com/dotnet/improving-csharp-memory-safety/?WT.mc_id=DT-MVP-5004827) は、かなり大きな方向転換の話です。

ここで面白いのは、`unsafe` を「ポインター構文を使うための印」としてではなく、**呼び出し側に安全性の責務があることを示す契約**として再設計しようとしている点です。  
さらに、安全コメントスタイルを導入して、どこにどういう注意が必要なのかをレビューしやすくしようとしています。

プレビューは .NET 11、正式化は .NET 12 想定とのことです。  
AI 補助コード生成が当たり前になっていく中で、「書ける」よりも「レビューできる」ことの重要性が上がっている、という背景もかなり納得感がありました。

普段 `unsafe` を使わない人でも、**C# が安全性をどう言語化しようとしているか**を知る意味で読んでおく価値があると思います。

### NuGet の pruning は、脆弱性レポートを見る人ほど効く

[NuGet Package Pruning: Cleaner Dependencies and Actionable Vulnerability Reports](https://devblogs.microsoft.com/dotnet/nuget-package-pruning-in-dotnet-10/?WT.mc_id=DT-MVP-5004827) は、地味に見えてかなり実務的です。

.NET 10 では `NuGetAuditMode=all` が既定になり、さらに platform-provided package を restore graph から pruning することで、**ノイズの多い依存関係や偽陽性の脆弱性警告を減らす**方向に進んでいます。

記事中では、transitive vulnerability report が 70% 減少したというテレメトリや、restore が最大 50% 改善するケースにも触れられています。  
このあたりは、CI で監査を回しているチームや、脆弱性レポートの読み解きに疲れている人ほど刺さる話です。

「脆弱性が増えた」のではなく、**本当に見たい警告を見やすくする**ための改善として読むと分かりやすいです。

### Process API の刷新は、日常的に使う人ほどうれしい

[Process API Improvements in .NET 11](https://devblogs.microsoft.com/dotnet/process-api-improvements-in-dotnet-11/?WT.mc_id=DT-MVP-5004827) は、個人的にかなり好きな更新です。

`Process.RunAndCaptureText[Async]`、`Run[Async]`、`StartAndForget`、`ReadAllText[Async]` など、今まで少し手間だったところがまとめて改善されています。  
特に、stdout / stderr を同時に drain しやすくして**デッドロックを避けやすい**のは、プロセス実行まわりで何度か苦労したことがある人には分かりやすい価値だと思います。

加えて、`KillOnParentExit`、`StartDetached`、`SafeProcessHandle` なども入り、Apple Silicon ではプロセス生成の高速化にも触れられています。

CLI ツール連携、ビルド補助、外部コマンド呼び出しを書く人は、かなり早めに見ておいてよさそうです。

### まとめ記事としては .NET 11 Preview 4 も便利

個別記事を読む前に全体をざっとつかみたいなら、[.NET 11 Preview 4 is now available!](https://devblogs.microsoft.com/dotnet/dotnet-11-preview-4/?WT.mc_id=DT-MVP-5004827) が入り口になります。

Runtime、SDK、Libraries、ASP.NET Core、MAUI、EF Core まで広く触れていて、今週の更新の見取り図として使いやすいです。  
上で触れた Process API 改善に加えて、MCP Server template の SDK 同梱、`dotnet watch` のモバイル改善、EF Core の approximate vector search などもここで把握できます。

「まず全体像を見て、その後で気になる個別記事に入る」という読み方に向いています。

### MAUI を見ている人は CoreCLR 移行を要チェック

[.NET MAUI Moves to CoreCLR in .NET 11](https://devblogs.microsoft.com/dotnet/dotnet-maui-moves-to-coreclr-in-dotnet-11/?WT.mc_id=DT-MVP-5004827) は、MAUI を使っている人なら優先度が高いです。

.NET 11 Preview 4 から Android / iOS / Mac Catalyst / tvOS で CoreCLR が既定になり、ランタイム統一が進みます。  
R2R、PGO、NativeAOT への道筋が見えやすくなるのは前向きですが、一方で複雑なアプリでは回帰報告もあるとのことでした。

この話は、期待だけで見るよりも、**自分のアプリで実測する前提で読む**のがよさそうです。  
必要なら `UseMonoRuntime` で一時回避できる点も、実務的には大事です。

### AI / MCP に関心があるならボーナス枠も面白い

AI や MCP に関心がある人向けには、[Announcing Agent Governance Toolkit MCP Extensions for .NET](https://devblogs.microsoft.com/dotnet/announcing-agent-governance-toolkit-mcp-extensions-for-dotnet/?WT.mc_id=DT-MVP-5004827) も押さえておくとよさそうです。

Public Preview のパッケージとして、policy enforcement、startup scanning、response sanitization などを 1 コールで導入できる、という整理でした。  
MCP サーバーを作るだけでなく、**運用時のガバナンスをどう載せるか**に関心がある人には、方向性をつかむ材料になりそうです。

## GitHub 側の重要更新

### 今週の GitHub は「既存運用に効く更新」が多い

GitHub 側は、新機能の数そのものより、**今のやり方を少し見直したくなる更新**が多かった印象です。

特に見ておきたいのは、npm、Issue 運用、Copilot の 3 つです。

### npm staged publishing は、公開フローを整えているチームほど影響が大きい

[Staged publishing and new install-time controls for npm](https://github.blog/changelog/2026-05-22-staged-publishing-and-new-install-time-controls-for-npm) は、npm を配布運用している人にはかなり重要です。

staged publishing が GA になり、CI/CD の trusted publishing と組み合わせやすくなりました。  
さらに npm 11.15.0 以降で `npm stage publish` が入り、`--allow-file`、`--allow-remote`、`--allow-directory` といった install-time controls も追加されています。

将来の v12 では `--allow-git` の既定が `none` に変わる予定とのことなので、**今のスクリプトや依存取得の前提がそのままでよいか**は早めに確認しておきたいところです。

### Issue fields は、ラベル運用を見直すきっかけになりそう

[Issue fields are now in public preview for all organizations](https://github.blog/changelog/2026-05-21-issue-fields-are-now-in-public-preview-for-all-organizations) は、地味に大きい更新です。

org レベルの typed metadata をすべての issue に付けられるようになり、Priority や Effort のような情報を、label ではなく **structured metadata** として扱えるようになります。  
検索、Projects 列、REST / GraphQL / webhook 連携もあるので、単なる UI 改善ではありません。

今までラベルで無理やり表していた情報を、もう少しきれいに整理できる可能性があります。  
Issue 運用が育っている組織ほど、「今のラベル設計を続けるか」を考える材料になりそうです。

### Copilot の Auto model selection は、既存ユーザーの使い方に直結する

[Auto model selection now routes based on your task in VS Code](https://github.blog/changelog/2026-05-20-auto-model-selection-now-routes-based-on-your-task-in-vs-code) は、Copilot を日常的に使う人ほど影響があります。

Copilot の Auto が、タスクに応じてモデルを選択するようになり、管理者ポリシーも尊重されます。  
しかも 1x 以下モデルに限定され、paid subscriber では premium requests が 10% 割引されるとのことです。

これは「新しいボタンが増えた」という話ではなく、**いつもの使い方の中身が変わる**話です。  
普段あまりモデルを意識せず使っている人ほど、挙動の違いを感じる場面が出てきそうです。

### Semantic issue search は、Issue を探す時間を減らしてくれそう

[Semantic issue search in Copilot Chat](https://github.blog/changelog/2026-05-20-semantic-issue-search-in-copilot-chat) は、Issue が多いリポジトリほど効きそうです。

exact match ではなく自然言語で Issue を探せるようになり、全 Copilot プランで GA とのことです。  
Issue triage や planning の初動で、「確か似た話があったはず」を拾いやすくなる可能性があります。

派手さはありませんが、日々の探し物のコストを下げる更新として見ておくとよさそうです。

### Copilot code review feedback の適用が、かなり実務寄りになってきた

[Easily apply Copilot code review feedback with Copilot cloud agent](https://github.blog/changelog/2026-05-19-easily-apply-copilot-code-review-feedback-with-copilot-cloud-agent) では、`Implement suggestion` が `Fix with Copilot` に変わり、適用先、モデル、追加指示をダイアログで選べるようになっています。

さらに、複数コメントの batch 修正も可能です。  
このあたりは、Copilot を「補完」ではなく**レビュー後の修正フローに組み込む**方向が強くなってきた、と見ると分かりやすいです。

レビューコメントをどう反映するかに時間がかかっていたチームには、試す価値がありそうです。

### 運用担当なら download URL の変更も確認しておきたい

[Copilot usage metrics reports now use GitHub-owned download URLs](https://github.blog/changelog/2026-05-20-copilot-usage-metrics-reports-now-use-github-owned-download-urls) は、管理者向けの補足ですが、対象の組織には重要です。

allowlist を使っている enterprise では、`copilot-reports.github.com` への切り替え確認が必要になります。  
開発者個人には直接見えにくい更新ですが、レポート取得が止まると後から困るので、管理側は早めに見ておくと安心です。
